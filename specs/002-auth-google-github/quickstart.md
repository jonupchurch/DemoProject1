# Quickstart: Authentication via Google & GitHub SSO

Validates the feature end-to-end once implemented. Assumes the tasks in `tasks.md` (generated by
`/speckit-tasks`) have been implemented — this is a validation guide, not an implementation guide.

## Prerequisites

- A Google OAuth client (Client ID + Secret) and a GitHub OAuth App (Client ID + Secret), each with
  a `http://localhost:3000/api/auth/callback/{provider}` redirect/callback URL registered.
- `.env` populated with `AUTH_SECRET`, `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`,
  `GITHUB_CLIENT_ID`, `GITHUB_CLIENT_SECRET` (already done — see conversation history).
- Prisma schema migrated for the new `User`/`Account`/`Session`/`VerificationToken` models:

```text
npx prisma migrate dev
```

## Run

```text
npm run dev
```

## Validation scenarios

Each maps directly to a user story's Independent Test in `spec.md`. Because this feature involves
real third-party OAuth consent screens, these are manual browser checks (research.md §6) rather than
automated end-to-end tests.

1. **Sign in with Google or GitHub (User Story 1)**
   - Visit `/` while signed out — confirm the sign-in prompt appears with both provider options.
   - Choose "Continue with Google," complete the real Google consent screen, confirm you land on
     `/decisions` with an empty list (a new account was created).
   - Sign out; repeat with "Continue with GitHub" using a GitHub account with a *different* email
     than the Google one — confirm a second, separate, empty decisions list.

2. **Decision data is private to the signed-in account (User Story 2)**
   - Signed in as the Google-created account, log a decision.
   - Sign out, sign in as the GitHub-created account — confirm the Google account's decision is
     NOT visible in this account's list.
   - Copy the Google account's decision detail URL, sign out, sign in as GitHub, and try to open
     that URL directly — confirm access is denied/redirected, not shown.
   - While fully signed out, attempt to visit `/decisions`, `/decisions/new`, and a decision detail
     URL directly — confirm every one redirects to the sign-in prompt.

3. **Linking by verified email (User Story 3)**
   - Using a Google account and a GitHub account that share the same, verified email address: sign
     in with Google first, log a decision, sign out, then sign in with GitHub.
   - Confirm you land in the *same* account (the previously logged decision is visible), not a new
     empty one.

4. **Sign out (User Story 4)**
   - While signed in, choose sign out — confirm you return to the signed-out sign-in prompt.
   - Use the browser back button to return to a previously visited decision page — confirm it
     redirects to the sign-in prompt rather than showing cached data.

## Non-functional checks

- Run Lighthouse against `/` (signed-out) in a production build — Performance and Accessibility
  should be consistent with phase 1's measurements (constitution Principle VII; phase 1's
  Performance gap is a documented, carried-forward exception, not something this phase needs to
  re-litigate).
- Run an automated accessibility check (axe) against the sign-in prompt — zero violations
  (constitution Principle IV).
- Confirm `src/proxy.ts` never imports anything from `src/auth.ts` or `@prisma/client` (would break
  the edge runtime) — this should also fail loudly at build time if violated, not just in review.
