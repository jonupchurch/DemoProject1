# Quickstart: Travel Photo Map

Validates the feature end-to-end once implemented. Assumes the tasks in `tasks.md` (generated by
`/speckit-tasks`) have been implemented — this is a validation guide, not an implementation guide.

## Prerequisites

- The existing local Postgres dev database, migrated with the new `Pin`/`Photo` tables and the
  `User.pins` back-relation (`data-model.md`):

```text
npx prisma migrate dev
```

- No new environment variables are required for local development — the local-filesystem
  `PhotoStorage` driver (research.md §2) needs no credentials. A production deployment additionally
  needs a Vercel Blob token, but that's out of scope for local validation.
- Dependencies installed (`npm install`), including the new map (`maplibre-gl` + its React binding)
  and any photo-upload-handling packages chosen during implementation.
- Signed in as at least two different accounts at some point during validation (to exercise
  cross-account ownership rejection) — reuse the existing JWT-session-crafting approach from
  earlier phases for automated checks, or two real browser sessions for manual validation.

## Run

```text
npm run dev
```

## Validation scenarios

Each maps directly to a user story's Independent Test in `spec.md`.

1. **Publish a pinned photo gallery (User Story 1)**
   - Signed in, open the travel map, choose a point, and upload one or more photos.
   - Confirm an explicit public-visibility notice appears before the pin can be submitted (FR-003).
   - Submit. Confirm the new pin now appears on the map with all of its photos and its coordinates.
   - Attempt to upload a non-image file or an oversized file — confirm it's rejected with a clear
     error and no pin is created (spec Edge Cases).
   - Signed out, attempt to reach the publish flow directly — confirm it redirects to sign-in first.

2. **Browse published pins publicly (User Story 2)**
   - Signed out, open the app. Confirm every existing pin's photos and location are visible with no
     sign-in prompt.
   - Open a pin with more than one photo — confirm all of its photos display together as a gallery,
     not just one cover image (FR-007).
   - Confirm a pin's location shows its raw latitude/longitude plus a working link to an external
     map service for that coordinate (FR-006).
   - Switch to the non-map list view — confirm the same pins/coordinates are reachable using only
     the keyboard (Tab/Enter, no mouse) and are announced sensibly by a screen reader (FR-008).

3. **Manage your own pins (User Story 3)**
   - As a pin's owner, edit its caption and location — confirm the change is visible to a
     signed-out view of the same pin.
   - Add a new photo to an existing pin's gallery, then remove a different photo (leaving at least
     one) — confirm both changes are reflected publicly.
   - Attempt to remove a pin's only remaining photo — confirm it's rejected (FR-010).
   - Signed in as a *different* account, open someone else's pin — confirm no edit/delete/photo
     controls are shown, and confirm directly invoking an edit/delete/add-photo/remove-photo action
     against that pin is rejected (FR-009, FR-011).
   - Delete a pin you own — confirm it and all of its photos no longer appear anywhere.

## Non-functional checks

- Run Lighthouse against the travel app's map view and a pin's gallery view in a production build
  (`npm run build && npm run start`) — Performance and Accessibility categories must each score 95+
  (constitution Principle VII), including the image-optimization requirements that principle now
  spells out for image-heavy views (next/image, responsive sizing, lazy-loading).
- Confirm the map view and the publish/edit flows are keyboard-operable and pass an automated
  accessibility check (e.g. `axe`) with no violations, and that the non-map list view independently
  passes the same check (constitution Principle IV).
- Confirm both light and dark color schemes render the map, galleries, and forms correctly with no
  illegible text/contrast issues (constitution Principle VI).
