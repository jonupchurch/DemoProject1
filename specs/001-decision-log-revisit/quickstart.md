# Quickstart: Core Decision Logging & Revisit/Resolve Flow

Validates the feature end-to-end once implemented. Assumes the tasks in `tasks.md` (generated by
`/speckit-tasks`) have been implemented — this is a validation guide, not an implementation guide.

## Prerequisites

- A Vercel Postgres database provisioned (or a local Postgres instance for local-only development),
  with both a pooled and a direct connection string available (see `research.md` §2).
- `.env` populated with `POSTGRES_PRISMA_URL` (pooled) and `POSTGRES_URL_NON_POOLING` (direct) —
  see `data-model.md` for the schema these apply to.
- Dependencies installed (`npm install`).
- Prisma schema migrated and the single `Owner` row seeded:

```text
npx prisma migrate dev
npx prisma db seed
```

## Run

```text
npm run dev
```

## Validation scenarios

Each maps directly to a user story's Independent Test in `spec.md`.

1. **Log a decision (User Story 1)**
   - Visit `/decisions`, start a new decision.
   - Enter a title, at least one named option, a confidence level, a category, and a review date.
   - Submit. Confirm the decision now appears in the `/decisions` list with a "Pending" indicator.
   - Repeat, omitting a required field — confirm the form rejects the submission and identifies the
     missing field (spec Edge Cases).

2. **Resolve a decision (User Story 2)**
   - Open a pending decision's detail page (`/decisions/[id]`).
   - Resolve it: pick a verdict, a satisfaction score, and enter learnings.
   - Confirm the decision's status becomes "Resolved" and its original entry (options, cost, risks,
     confidence, category) is no longer editable.
   - Edit the resolution's verdict/satisfaction/learnings — confirm the change saves and the
     original entry remains untouched.

3. **Browse decision history (User Story 3)**
   - With a mix of pending and resolved decisions logged, confirm `/decisions` lists all of them
     regardless of status, and each opens to its full detail.

4. **Edit or delete a decision (User Story 4)**
   - Edit a pending decision's fields; confirm changes persist.
   - Delete a decision (either status); confirm a confirmation prompt appears and, once confirmed,
     the decision no longer appears anywhere.

## Non-functional checks

- Run Lighthouse against `/decisions` and `/decisions/[id]` in a production build
  (`npm run build && npm run start`) — Performance and Accessibility categories must each score 95+
  (constitution Principle VII).
- Confirm all forms are operable via keyboard alone and pass an automated accessibility check
  (e.g. `axe`) with no violations (constitution Principle IV).
